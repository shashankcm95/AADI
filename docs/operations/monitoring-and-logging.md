# Monitoring and Logging

This document describes how the Arrive platform produces, structures, and centralizes operational telemetry. It covers the structured logging framework, log field semantics, CloudWatch integration, correlation ID tracking, log levels, health endpoints, key operational metrics, and guidance for setting up CloudWatch alarms.

---

## Structured Logging Approach

Every Lambda function in the Arrive backend produces structured JSON logs using a custom logging framework defined in `services/shared/python/shared/logger.py`. This framework wraps Python's standard `logging` module with a JSON formatter and a structured logger adapter, producing one JSON object per log line. The design was chosen so that CloudWatch Logs Insights can parse and query log fields directly without custom parsing rules.

The logging framework consists of three main components. The `JSONFormatter` class formats every log record as a single-line JSON object. The `StructuredLogger` class is a `logging.LoggerAdapter` that injects bound context fields into every log record it produces. The `get_logger` factory function is the public entry point that services use to obtain a configured logger instance.

To use the logger in any Lambda handler or utility module, import the factory and create a logger:

```python
from shared.logger import get_logger

log = get_logger("orders.customer")
log.info("order_created", extra={"order_id": "ord_abc", "restaurant_id": "rest_xyz"})
```

The logger supports a `bind()` method that returns a new logger instance with additional context fields permanently attached. This is useful for binding a correlation ID at the start of a request and having it appear in every subsequent log line:

```python
log = log.bind(correlation_id=request_id, customer_id=customer_id)
log.info("processing_order")  # correlation_id and customer_id appear automatically
```

## Log Fields

Every log line produced by the framework contains a consistent set of base fields, plus any custom fields passed via the `extra` parameter or bound context.

**timestamp** is the ISO 8601 formatted time when the log record was created. This is generated by Python's `logging.Formatter.formatTime` and reflects the Lambda execution environment's clock.

**level** is the log severity level as a string: `DEBUG`, `INFO`, `WARNING`, `ERROR`, or `CRITICAL`. This directly maps to Python's standard logging levels.

**logger** is the dotted name of the logger instance, such as `orders.customer` or `pos.auth`. This field identifies which module or subsystem produced the log entry and is useful for filtering in CloudWatch Insights.

**message** is the human-readable log message. In Arrive's convention, messages use snake_case identifiers (e.g., `order_created`, `api_key_lookup_error`) rather than natural language sentences. This makes them reliably searchable and parseable.

**service** identifies the service that produced the log entry. This defaults to the value of the `SERVICE_NAME` environment variable, or `arrive` if the variable is not set. In production, each Lambda function should set `SERVICE_NAME` to its service name (e.g., `orders`, `restaurants`, `users`, `pos`).

**correlation_id** is extracted from the API Gateway request context using the `extract_correlation_id` helper function. It reads `requestContext.requestId` from the API Gateway event, falling back to the `x-amzn-requestid` header, or `no-correlation-id` if neither is available. This field ties all log entries for a single API request together.

Beyond these base fields, any additional key-value pairs passed as `extra` appear as top-level fields in the JSON output. Common contextual fields include `order_id`, `restaurant_id`, `customer_id`, `status`, `elapsed_ms`, and `error`.

If an exception is being logged (via `log.error("message", exc_info=True)`), the `exception` field contains the formatted traceback string.

## Finding Logs in CloudWatch

Each Lambda function writes to its own CloudWatch Log Group, following the naming pattern `/aws/lambda/<function-name>`. Because Arrive uses SAM, the function names include the stack name prefix. For example, the Orders function's log group might be `/aws/lambda/arrive-dev-OrdersFunction-ABC123`.

To query logs across all services, use CloudWatch Logs Insights with a query that spans multiple log groups. The JSON structure of Arrive's logs means you can use Insights' built-in JSON parsing:

```
fields @timestamp, level, logger, message, correlation_id, order_id
| filter level = "ERROR"
| sort @timestamp desc
| limit 100
```

To find all log entries for a specific order:

```
fields @timestamp, level, message, correlation_id
| filter order_id = "ord_abc123"
| sort @timestamp asc
```

To find all log entries for a specific API request across services, query across all service log groups using the correlation ID:

```
fields @timestamp, level, logger, message
| filter correlation_id = "abcdef-1234-5678"
| sort @timestamp asc
```

The SAM template configures Lambda logging with `LogFormat: JSON` and both `ApplicationLogLevel` and `SystemLogLevel` set to `INFO` in the `Globals.Function.LoggingConfig` section. This means Lambda's own platform logs (START, END, REPORT) are also emitted in JSON format alongside the application's structured logs.

## Correlation ID Tracking

Correlation IDs are the primary mechanism for tracing a request across log entries within a single service. When an API Gateway HTTP API receives a request, it assigns a unique `requestId` in the `requestContext`. The `extract_correlation_id` function in `shared/logger.py` extracts this ID, and handlers bind it to their logger instance at the start of each invocation.

A typical Lambda handler pattern looks like this:

```python
from shared.logger import get_logger, extract_correlation_id

log = get_logger("orders")

def lambda_handler(event, context):
    correlation_id = extract_correlation_id(event)
    log_ctx = log.bind(correlation_id=correlation_id)
    log_ctx.info("request_received")
    # ... handler logic, all using log_ctx ...
```

Every subsequent log call through `log_ctx` automatically includes `correlation_id` in the JSON output. This means you can take any error log line, extract its `correlation_id`, and find every log entry related to that request.

For cross-service tracing (for example, when the Orders service reads from the Restaurants config table), AWS X-Ray active tracing is enabled globally in the SAM template (`Tracing: Active` in Globals and `TracingEnabled: true` on the API). X-Ray traces provide a service map and latency breakdown across Lambda invocations and AWS SDK calls.

## Log Levels and Their Usage

The Arrive codebase follows consistent conventions for when each log level is used.

**INFO** is the default production level. It is used for significant business events: order creation, status transitions, API key validation success, user profile updates, restaurant creation, and health check responses. These events represent normal system operation and are the primary source of operational visibility.

**WARNING** is used for conditions that are unusual but not erroneous. Examples include missing environment variables (such as `POS_API_KEYS_TABLE` not being set), fallback behavior being triggered (such as using a default CORS origin), legacy users without a custom role attribute, and approaching capacity limits.

**ERROR** is used for failures that affect a specific request but do not represent a system-wide problem. API key lookup failures, DynamoDB operation exceptions, invalid input that passes initial validation but fails business rules, and unhandled exceptions in request processing all produce ERROR entries.

**DEBUG** is not used in production by default. The log level is controlled by the `LOG_LEVEL` environment variable, which defaults to `INFO`. To enable debug logging for a specific function, set `LOG_LEVEL=DEBUG` in its environment variables. Debug logs include detailed request/response payloads and internal state during complex operations like the order state machine.

**CRITICAL** is reserved for conditions that indicate a fundamental system failure, such as an inability to connect to DynamoDB or a corrupted configuration table. These are rare in practice.

## Health Endpoints

Two services expose unauthenticated health check endpoints that return a simple JSON response indicating the service is operational.

The **Users service** exposes `GET /v1/users/health` with `Auth: Authorizer: NONE`, meaning it does not require a JWT token. This endpoint is suitable for external monitoring tools, load balancer health checks, or uptime services.

The **Restaurants service** exposes `GET /v1/restaurants/health` as a similar unauthenticated endpoint.

The Orders service does not currently have a dedicated health endpoint, but its authenticated endpoints can serve as synthetic monitors when called with a valid token.

Health endpoints return CORS headers and a 200 status code with a JSON body. They verify that the Lambda function is running and can respond to HTTP requests. They do not perform deep dependency checks (such as verifying DynamoDB connectivity), keeping them fast and reliable.

## Operational Metrics to Watch

Beyond application logs, several AWS-level metrics provide insight into system health.

**Lambda invocation metrics** include `Invocations`, `Errors`, `Throttles`, and `Duration`. Watch `Errors` for any function exceeding a low threshold, and `Duration` approaching the 15-second timeout configured in the SAM globals. Throttles indicate that concurrent execution limits have been reached.

**API Gateway metrics** include `4xx` and `5xx` error rates, `Latency`, and `Count`. A spike in `4xx` errors may indicate client misconfiguration or an authorization issue. A spike in `5xx` errors indicates backend failures.

**DynamoDB metrics** include `ConsumedReadCapacityUnits`, `ConsumedWriteCapacityUnits`, `ThrottledRequests`, and `SystemErrors`. Because all tables use on-demand billing, throttling should be rare, but it can occur during sudden traffic spikes that exceed the table's auto-scaling rate.

**CloudFront metrics** include `Requests`, `BytesDownloaded`, `4xxErrorRate`, and `5xxErrorRate`. A high `4xxErrorRate` on the frontend distributions may indicate stale cache entries or missing assets after a deployment without cache invalidation.

**SQS metrics** for the PostConfirmationDLQ and GeofenceResyncDLQ should be monitored for `ApproximateNumberOfMessagesVisible`. Any messages in these dead-letter queues indicate failed processing that needs investigation.

## Setting Up CloudWatch Alarms

For production deployments, configure CloudWatch alarms on the following metrics.

Create an alarm on each Lambda function's `Errors` metric with a threshold of 1 or more errors in a 5-minute period. This catches any backend failures quickly. Route the alarm to an SNS topic that delivers to your on-call channel (email, Slack, PagerDuty).

Create an alarm on the PostConfirmationDLQ's `ApproximateNumberOfMessagesVisible` metric with a threshold of 1 or more. Messages in this queue mean new users are not being properly initialized with the `customer` role, which will cause authorization failures for those users.

Create an alarm on the GeofenceResyncDLQ (in the Restaurants stack) with the same threshold. Messages here indicate that restaurant geofence synchronization has failed, which could cause missed arrival events.

Create an alarm on each API Gateway's `5xx` metric with a threshold appropriate for your traffic level. For low-traffic environments, 5 errors in 5 minutes is a reasonable starting point.

Create an alarm on the POS HTTP API's `Throttle` metric if POS integration is enabled. Throttled requests mean POS systems are exceeding the 50 requests-per-second rate limit, and you may need to adjust the limit or work with the POS vendor to implement backoff.

For the `ExpireOrdersFunction`, which runs on a 5-minute schedule, create an alarm on its `Errors` metric. This function is responsible for expiring abandoned orders; if it fails, stale orders will accumulate and consume capacity slots.

Consider using CloudWatch Logs Insights scheduled queries to generate metric filters for application-level events. For example, a metric filter counting log entries where `message = "api_key_lookup_error"` can power an alarm that detects POS authentication infrastructure failures.
