# Avatar Access Policy Decision

Date: 2026-03-02  
Finding: F-008 (Reassess avatar public-read policy)

## Decision
User avatars are now **private objects** in S3. Read access is provided through short-lived **presigned GET URLs** generated by the `users` service.

## Why
- Public-read avatar objects made user media enumerable and permanently cacheable by anyone with object URLs.
- Private-by-default aligns with least privilege and reduces accidental data exposure blast radius.
- Presigned reads keep direct client rendering simple while preserving access control at the API boundary.

## Implementation
- Removed anonymous `s3:GetObject` bucket policy from `services/users/template.yaml`.
- Enabled strict public access block settings (`BlockPublicPolicy=true`, `RestrictPublicBuckets=true`).
- `GET /v1/users/me` and `PUT /v1/users/me` now return:
  - `picture`: canonical S3 key (`avatars/<user>-<ts>.<ext>`)
  - `picture_url`: presigned read URL (short-lived)
- `POST /v1/users/me/avatar/upload-url` continues to return a presigned upload URL and `s3_key`; no public URL is emitted.
- Web/mobile clients now update profile picture using `s3_key` and render `picture_url`.

## Operational Notes
- Presigned read URL TTL is controlled by `AVATAR_GET_URL_TTL_SECONDS` (default: 900 seconds).
- Existing records storing bucket URLs are normalized to key format when possible.
